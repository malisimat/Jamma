Jamma — VST Channel Routing Reference

How audio channels are defined, negotiated, and routed through the Station → LoopTake → Loop hierarchy and their VST insert chains.

2. Station — Channel Count

How it is defined

Station has a bus channel count that drives all its internal buffer, mixer, and VST scratch allocation:

// Station.cpp
const unsigned int Station::_DefaultNumBusChannels = 8;

Station::Station(...)
{
    SetNumBusChannels(_DefaultNumBusChannels); // called in constructor
}

SetNumBusChannels(chans) rebuilds:

  • _audioBuffers — one ring-buffer per bus channel (latency-compensated mix-down target).
  • _audioMixers — one AudioMixer per bus channel (routes to DAC outputs via routing matrix).
  • _vstBlockScratch / _vstBlockPtrs — flat scratch storage and pointer array, sized chans * MaxBlockSize.

It is called again whenever the user changes the channel count via the GuiRack UI panel, and also propagated to all child LoopTakes via take->SetNumBusChannels(chans).

WriteBlock (DAC output path)

Each audio block the station plays back:

  1. Each LoopTake writes its mix-down samples into the Station’s _audioBuffers via OnBlockWriteChannel.
  2. Station reads each buffer into the corresponding _vstBlockPtrs[i] scratch slot and applies masterLevel.
  3. The VST chain is called: _vstChain->ProcessBlockMulti(_vstBlockPtrs.data(), channelCount, sampsToRead) where channelCount = min(_audioBuffers.size(), _audioMixers.size()).
  4. Each processed channel is forwarded to the DAC via its _audioMixers[i]->WriteBlock().

3. LoopTake — Channel Count

How it is defined

A LoopTake holds one Loop per recorded input channel. Its channel count is therefore dynamic — it grows as loops are added during recording:

// LoopTake.cpp — NumInputChannels
unsigned int LoopTake::NumInputChannels(Audible::AudioSourceType source) const
{
    return (_changesMade && _flipLoopBuffer) ?
        (unsigned int)_backLoops.size() :
        (unsigned int)_loops.size();  // one Loop per input channel
}

NumOutputChannels for AUDIOSOURCE_LOOPS/AUDIOSOURCE_MIXER returns NumBusChannels() (the station bus size, propagated from parent Station). For ADC/MONITOR it also returns the loop count.

Scratch buffer sizing

Whenever loops are committed in _CommitChanges(), _ResizeVstScratch(_audioBuffers.size()) is called, keeping _vstBlockPtrs in sync with the loop count:

// LoopTake.cpp — _CommitChanges
_loops       = _backLoops;
_audioBuffers = _backAudioBuffers;
_audioMixers  = _backAudioMixers;
_ResizeVstScratch((unsigned int)_audioBuffers.size());

WriteBlock (DAC output path)

Each Loop in the take reads its mono audio into the corresponding _audioBuffers[i] via _audioMixers[i]->WriteBlock. The LoopTake then:

  1. Copies each buffer into _vstBlockPtrs[i].
  2. Calls chain->ProcessBlockMulti(_vstBlockPtrs.data(), channelCount, sampsToRead) where channelCount = min(_audioBuffers.size(), _audioMixers.size()).
  3. Passes each processed channel to its output AudioMixer (which routes it to the Station bus).
Note: The guard _vstBlockPtrs.size() >= channelCount (checked before the ProcessBlockMulti call) is a runtime safety check to ensure the scratch array is large enough. Since _ResizeVstScratch always sets it to _audioBuffers.size(), this should always pass in normal operation.

4. Loop — Channel Count

Loop is always mono. It records a single stream from one ADC input channel (identified by LoopParams::Channel) into a single BufferBank. There is no multichannel concept at the Loop level.

Its AudioMixer can spread the mono signal to one or more output channels (wire or pan behaviour), but the raw loop audio itself is always a single float stream.

// Loop.cpp — WriteBlock reads mono into tempBuf, then calls VST
float tempBuf[constants::MaxBlockSize];
auto sampsToWrite = ReadBlock(tempBuf, sampOffset, numSamps);

if (sampsToWrite > 0)
{
    if (_vstChain && _vstChain->IsActive())
        _vstChain->ProcessBlock(tempBuf, static_cast<int>(sampsToWrite));

    _mixer->WriteBlock(dest, tempBuf, sampsToWrite);
}

5. VST Plugin Configuration at Each Level

5.1 Station — VST load

// Station.cpp — JOB_LOADVST handler
auto hostChannels = NumBusChannels();    // e.g. 8
if (hostChannels == 0u) hostChannels = 1u;
plugin->Load(action.VstPath, _sampleRate, _blockSize,
             hostChannels, vst::HostedLayoutMode::Exact);

The plugin is loaded with exactly the current bus channel count and HostedLayoutMode::Exact. This means:

  • setBusArrangements requests a speaker arrangement matching hostChannels (e.g. kStereo for 2, k71Music for 8).
  • After negotiation, IsHostedLayoutCompatible requires that the plugin’s actual channels after negotiation are ≤ hostChannels. This allows a stereo plugin (2 ch) to load on an 8-bus station; it rejects a plugin that insists on more channels than the station offers.

At runtime, ProcessBlockMulti is called with numChannels = channelCount (the live bus size). The updated guard numChannels < requestedChannels ensures:

  • If the bus grew ≥ requestedChannels: the plugin processes its channels; extra channels pass through untouched.
  • If the bus shrank below requestedChannels: the plugin call is a silent no-op.

5.2 LoopTake — VST load

// LoopTake.cpp — JOB_LOADVST handler
auto hostChannels = NumInputChannels(Audible::AUDIOSOURCE_LOOPS);  // = _loops.size()
if (hostChannels == 0u) hostChannels = 1u;
plugin->Load(action.VstPath, _sampleRate, _lastBufSize,
             hostChannels, vst::HostedLayoutMode::Exact);

The plugin is loaded with the current loop count as the channel count. Since NumInputChannels ignores the source type and always returns the loop count, this is straightforward:

  • A take with 4 loops loads its VST as a 4-channel plugin.
  • A stereo plugin loaded on that 4-loop take will process loops 0–1 and leave loops 2–3 untouched (after the fix to ProcessBlockMulti).

5.3 Loop — VST load

// Loop.cpp — JOB_LOADVST handler
plugin->Load(action.VstPath, _sampleRate, _blockSize,
             1u, vst::HostedLayoutMode::MonoFlexible);

Always mono, always MonoFlexible.

  • ResolveHostedBusLayout(MonoFlexible, 1, inputBusChans, outputBusChans) uses the plugin’s own natural bus width rather than insisting on 1.
  • IsHostedLayoutCompatible under MonoFlexible always returns true (any positive channel count accepted).
  • At runtime: CopyMonoToInputBuffers broadcasts the mono buffer to all N plugin input channels. FoldOutputToMono averages all M plugin output channels back to mono.
  • A 6-channel plugin on a mono loop source is perfectly legal and works correctly.

6. Channel Mixing: What Happens at Each Width Combination

6.1 At the Loop level (MonoFlexible)

Plugin channelsInput pathOutput path
1 (mono) Mono buffer copied to channel 0 Channel 0 written back to mono output
2 (stereo) Mono buffer broadcast to ch 0 and ch 1 (CopyMonoToInputBuffers) Ch 0 + ch 1 averaged ÷ 2 back to mono (FoldOutputToMono)
N > 1 Mono buffer broadcast to all N channels All N channels summed and divided by N back to mono
Fold-to-mono note: FoldOutputToMono uses a simple equal-weight average. For a reverb or stereo-widener this collapses both sides into centre, which is correct behaviour for a mono loop source.

6.2 At the Station / LoopTake level (Exact, ProcessBlockMulti)

The key functions are CopyMultiToInputBuffers and CopyOutputToMulti:

CopyMultiToInputBuffers

Copies source channel i to plugin input channel i for i = 0 … inputChannels-1. Source channels beyond the plugin width are simply not read. Plugin input channels beyond the source width are zero-filled.

CopyOutputToMulti

Copies plugin output channel i to destination channel i for i = 0 … min(outputChannels, destChannels)-1. Destination channels beyond the plugin’s output width are left untouched (pass-through). No mixing occurs.

Host bus N Plugin width P Input to plugin Output from plugin Channels N > P
8 8 (exact match) All 8 host channels All 8 channels written back N/A
8 2 (stereo plugin) Host channels 0–1 copied to plugin inputs Plugin outputs 0–1 written back to host channels 0–1 Pass-through unchanged
8 1 (mono plugin) Host channel 0 copied to plugin input 0 Plugin output 0 written back to host channel 0 Channels 1–7 pass through
5 6 (wider than host) Load fails — IsHostedLayoutCompatible rejects actual channels > requested. Plugin is not activated.
1 2 (stereo plugin) ProcessBlockMulti silent no-op — numChannels (1) < requestedChannels (2). Guard rejects the call.

6.3 The “5-channel source, 6-channel VST” question

With HostedLayoutMode::Exact (Station and LoopTake):

  1. Host requests numChannels = 5. ResolveHostedBusLayout(Exact, 5, …) yields { requested=5, input=5, output=5 }.
  2. setBusArrangements asks the plugin to accept a 5-channel speaker arrangement.
  3. getBusArrangement reads back what the plugin actually agreed to.
  4. IsHostedLayoutCompatible checks actualInputChannels ≤ 5 && actualOutputChannels ≤ 5.
  5. If the plugin insists on 6 channels regardless: 6 ≤ 5 is false → Load() returns false. The plugin is not added to the chain.
  6. If the plugin negotiates down to 5 (or fewer): loading succeeds, and it operates as a 5-channel (or narrower) plugin on the 5-channel bus.
Practical note: Most professional VST3 plugins honour setBusArrangements and will negotiate to any requested count up to their maximum. A plugin that hard-codes 6 channels and cannot be configured for fewer will simply fail to load on a 5-bus station. The user would need to set the station to 6 (or more) bus channels first, then load the plugin.

7. Dynamic Channel Count Changes

Channel counts at all three levels can change after plugins are loaded. The table below summarises what happens in each case.

Event What is updated Is the VST reloaded? Resulting behaviour
Station bus channel count changed (user adjusts GuiRack) _audioBuffers, _audioMixers, _vstBlockScratch, _vstBlockPtrs all resized.
Child LoopTakes notified via SetNumBusChannels.		 No — loaded VSTs retain their frozen requestedChannels. New bus ≥ requestedChannels: plugin processes its channels, excess pass through.
New bus < requestedChannels: ProcessBlockMulti guard fires; plugin is silently skipped.
New Loop recorded in a LoopTake _backLoops, _backAudioBuffers grow by one. _CommitChanges calls _ResizeVstScratch to grow _vstBlockPtrs. No New loop count ≥ requestedChannels: plugin processes its channels, new loop channel passes through.
New loop count < requestedChannels: plugin silently skipped (no regression from original behaviour).
New VST loaded on a Station or LoopTake New plugin loaded on the job thread with the current channel count. Swapped in via _backVstChain/_flipVstChain. N/A (fresh load) Plugin’s requestedChannels is set to the current bus/loop count. All normal routing applies.
VstChain with multiple plugins, mixed widths Each plugin in the chain is called sequentially by VstChain::ProcessBlockMulti. N/A Each plugin applies its own guard independently. A mono plugin → stereo plugin → 8-channel plugin chain on an 8-bus station: the mono plugin modifies channel 0 only, the stereo plugin then sees the updated channels 0–1, and the 8-channel plugin sees all 8. Channels not touched by a plugin retain the values from the previous stage.

The mismatch risk

Known gap: There is no mechanism to automatically reload (or reload-and-renegotiate) a loaded VST plugin when the Station bus count or LoopTake loop count changes after the plugin was loaded. The plugin’s requestedChannels is frozen at load time. The immediate consequence:
  • Station bus shrinks below the plugin’s channel count → plugin is silently bypassed until bus grows back.
  • LoopTake loop count shrinks below the plugin’s channel count → same silent bypass.
If the user needs the plugin to work at a different channel width they must unload and reload it after the channel count change.

8. Key Helper Functions (VstAudioBuffers.h)

Function Signature Behaviour
CopyMonoToInputBuffers (monoInput, numSamples, inputChannels, inputChannelBuffers) Broadcast: copies mono buffer into every plugin input channel. Used by ProcessBlock.
CopyMultiToInputBuffers (sourceChannelBuffers, sourceChannels, numSamples, inputChannelBuffers, inputChannels) Copies source channel i → plugin input i, for i < min(source, input). Plugin inputs beyond source width are zero-filled.
FoldOutputToMono (outputChannelBuffers, outputChannels, numSamples, monoOutput) Averages all plugin output channels into one mono buffer (equal-weight mix). Used by ProcessBlock.
CopyOutputToMulti (outputChannelBuffers, outputChannels, numSamples, destinationChannelBuffers, destinationChannels) Copies plugin output i → destination i for i < min(output, dest). Destination channels beyond the plugin width are left untouched (pass-through). Used by ProcessBlockMulti.
ResolveHostedBusLayout (mode, requestedChannels, inputBusChannels, outputBusChannels) Exact: forces requested × requested. MonoFlexible: uses the plugin’s own natural bus sizes.
IsHostedLayoutCompatible (layout, actualInputChannels, actualOutputChannels) Exact: requires actual ≤ requested (allows narrower plugin). MonoFlexible: always true.

9. ProcessBlockMulti Guard Logic (after fix)

// VstPlugin.cpp — ProcessBlockMulti
// Old: numChannels != _impl->requestedChannels  → exact match only
// New: numChannels <  _impl->requestedChannels  → allow caller to be wider
if ((numChannels <= 0) || (numChannels < _impl->requestedChannels)
    || (numSamples <= 0) || (static_cast<unsigned int>(numSamples) > constants::MaxBlockSize))
    return;

Consequences of the change:

Condition Old behaviour New behaviour
numChannels == requestedChannels Processes normally Processes normally
numChannels > requestedChannels (wider bus than plugin) Silent no-op Plugin processes its channels; excess pass through
numChannels < requestedChannels (bus shrank) Silent no-op Silent no-op (unchanged)

The pass-through behaviour for extra channels is guaranteed by CopyOutputToMulti, which only writes min(outputChannels, numChannels) channels back and leaves the rest of the destination array untouched.
