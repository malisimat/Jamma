VSTi Instrument Support Review

Implementation notes for station-owned VST instruments receiving live and recorded MIDI.

Scope Station-owned VST2/VST3 instruments, live MIDI fanout, and recorded MIDI loop playback.
Persistence No store or restore changes were made.
UI No UI changes were made; routing is exposed at the engine/API level.
Tests Build and full native suite pass.

What Changed

Stations now act as the instrument host point. A station can own a VST chain, receive live MIDI from the shared MIDI ingress, receive recorded MIDI events from its loop takes, and process those MIDI events before the station bus is rendered through the VST chain.

  • IVstPlugin and VstChain gained block-scoped MIDI delivery methods.
  • Station gained live MIDI ingress, MIDI-output-to-plugin routing, and recorded MIDI playback dispatch.
  • LoopTake can read all MIDI loops in a block through indexed output sinks.
  • Scene::_PumpMidi fans channel voice input to all non-remote stations so one MIDI input can play multiple station instruments.
  • Scene::_OnAudio passes the absolute block start to stations for sample-accurate VST MIDI offsets.
  • VST2 instruments receive fixed-capacity VstMidiEvent blocks via effProcessEvents.
  • VST3 instruments receive fixed-capacity IEventList note events through ProcessData.inputEvents.

Routing Model

With no configured MIDI VST routes, station MIDI fans out to every plugin in that station's VST chain. When routes exist, a matching MIDI output index is delivered to the selected plugin index; unmatched outputs keep the default fanout behavior.

Live MIDI uses Station::LiveMidiOutputIndex. Recorded MIDI outputs are indexed by walking the station's loop takes and the MIDI loops within each take, preserving deterministic output indexes without changing file storage.

Real-Time Review

Rubber-duck question: does the audio callback allocate, lock, block, or touch slow I/O in the new MIDI/VST path? Answer: no new callback-owned path does that.
  • Live MIDI ingress uses the existing lock-free SPSC queue.
  • Recorded MIDI dispatch uses stack-only sink adapters.
  • VST2 MIDI event storage is a fixed array capped at 256 events per block.
  • VST3 MIDI event storage is a fixed array capped at 256 events per block.
  • Route publication allocates outside the audio callback and is consumed via atomic shared snapshots.
  • No new mutexes, blocking calls, filesystem calls, logging, or dynamic allocation were added to Scene::_OnAudio, Station::WriteBlock, LoopTake::ReadMidiBlock, or the VST process path.

Coverage Added

  • Live MIDI reaches a station VST plugin.
  • The same live MIDI input can play multiple stations.
  • Live MIDI can route to a specific plugin index.
  • Recorded MIDI loop playback can route to a specific station plugin.

Known Limits

  • Tests use fake plugins and do not load an actual VSTi, by design.
  • VST3 MIDI event support currently maps note on and note off events. Other controller-style input can be added once host/plugin expectations are pinned down.
  • Routing is present in engine APIs. No visible UI or persistence was added, matching the plan's explicit constraints.
