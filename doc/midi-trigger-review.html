Jamma MIDI implementation review

Multi-device MIDI is now routed, recorded, and tested.

The implementation moved from one global MIDI input to multiple callback-owned endpoints. Trigger activation is device-specific, and MIDI loop recording can subscribe to several devices on the same channel without collapsing them together.

What Changed

Config

Required device list

user.midi.devices now declares enabled MIDI inputs. The legacy single-device shape is rejected by parser coverage.

Routing

Slot-based trigger dispatch

Each opened MIDI input gets a stable slot. Trigger routes resolve device names to slots before dispatch.

Recording

Device-aware MIDI loops

midiinputdevices lets loop recording capture the same channel from multiple devices as separate streams.

Callback safety

One producer per queue

Each endpoint owns its own SPSC queue, preserving the queue model without locks in the MIDI callback path.

JSON

String arrays keep spaces

The shared JSON array tokenizer now preserves names such as MPK mini and Aux Keys.

Cleanup

Stale compatibility removed

The old slot-zero wrapper and shared-main MIDI trigger state were removed from Scene.

Risk Review

Still worth hardware testing

Real device lifecycle

Unit tests cover routing and queue behavior. Actual hardware should still be used to verify open, close, re-init, and active callback teardown across multiple MIDI drivers.

Exact names

Device matching is literal

Configured names must exactly match active MIDI names. Startup logging now catches unresolved trigger and loop-record devices, but there is no fuzzy matching.

Architecture

Hot paths stay quiet

The callback path pushes into a preallocated SPSC queue. Logging, route resolution, and validation happen outside the real-time callback path.

Cleanup

Generic parsing moved down

Generic JSON field helpers now live with Json, keeping RigFile focused on rig semantics.
