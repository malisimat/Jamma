Jamma · Change Brief

Fixing the VST2 Editor:
repaint, input & clean shutdown

What changed on this branch and why. The headline result: hosted VST2 plugin editors (notably Native Instruments Battery 4) now repaint continuously and take input, instead of painting one frozen frame — without turning the rest of the app white or hanging on exit.

feature/play-xp → bugfix/vst2-editor

01The headline bug

A hosted VST2 editor painted exactly once, then froze forever.

When a VST2 plugin with an editor was opened inside Jamma, the editor drew its first frame correctly and then went completely static — meters, clocks and animated controls all frozen — even though the plugin was clearly alive and spamming host callbacks. The reference Steinberg minihost sample hosts the same plugin and repaints continuously, so that became the behavioural target.

Root cause: thread affinity between effOpen and effEditOpen

The decisive cause was that the plugin was instantiated and opened on a different thread from the one that drove its editor. Originally VSTPluginMain() / effOpen ran on Jamma's job thread, while effEditOpen / effEditIdle ran on the UI thread. Many plugins (NI-style especially) bind their GUI/idle dispatch to the thread that instantiated the effect. With that split, the editor's repaint pump was orphaned: it built and painted once, then never fired again. It was never an idle-cadence problem — even at ~50 Hz the editor stayed frozen because the idle calls were crossing a thread boundary the plugin didn't expect.

02The core fixes

FIX Instantiate on the UI thread

Vst2Plugin::_InstantiateEffect()

VSTPluginMain() + effOpen + name/channel setup moved into PreInit() on the UI thread, so the effect and its editor share one thread. Load() (job thread) only does sample-rate/block-size/resume, and instantiates as a fallback only if PreInit was skipped.

FIX Parent directly into the frame

VstEditorWindow::Create()

The plugin is parented straight into the editor frame HWND (minihost convention), so host-callback posts (audioMasterSizeWindow, idle, updateDisplay) land on a window proc that actually handles them, instead of being silently dropped by an intermediate STATIC.

FIX Per-thread process level

audioMasterGetCurrentProcessLevel

Answer kVstProcessLevelRealtime only when the query arrives on the audio thread (stamped via _audioThreadId in ProcessBlock*); otherwise answer kVstProcessLevelUser. Reporting realtime on the GUI thread makes many plugins skip repaints.

GUARD OpenGL context restore

Vst2Plugin::GlContextScope (RAII)

Some plugins make their own GL context current during effOpen/effEditOpen/idle and never restore Jamma's. The guard snapshots wglGetCurrentContext() / DC and restores them on scope exit, preventing the "whole app paints white / framebuffer incomplete" regression.

Why the GL guard matters: moving instantiation onto the UI thread (which owns the OpenGL render context) introduced a second, closely-related failure — context theft turning the entire app white on plugin load. The GlContextScope guard wraps every UI-thread dispatch into plugin code and is a no-op on the job-thread fallback path (which has no GL context).

Editor host-callback contract, completed

Idle & child-window tracking

  • Replaced the global WH_CALLWNDPROCRET mouse-move hook with direct child parenting plus a 20 ms (~50 Hz) WM_TIMER idle, matching minihost cadence.
  • VstEditorWindow now tracks the plugin's child HWND (_pluginChildWnd) via enum-before/after-open diffing, used for resize and focus routing.
  • WM_SETFOCUS forwards keyboard focus into the plugin child window.

03Stability: input, shutdown & threading

Fixes that surfaced once the editor actually worked.

FIX Stuck modifier keys on editor focus

Key-up events go to whichever window has focus. When focus moved to a VST editor, modifier key-ups (Ctrl/Shift/Alt) never reached Jamma, leaving the modifier bitmask stuck (e.g. a phantom Ctrl-quantisation overlay). Window::ClearModifiers() on WM_KILLFOCUS now dispatches synthetic key-ups through the normal action path so scene state unwinds cleanly.

FIX Clean shutdown with editors open

Added ForceUnloadAllVstPlugins() down the full ownership chain (Scene → Station → LoopTake → Loop), draining active and back VST chains into the UI-thread destroy queue deterministically. The quit path closes editors, force-unloads plugins, drains the queue, then closes audio — no more hang or teardown crash on exit after opening a plugin editor.

THREADING Immutable snapshot access

Cross-thread reads of loop-takes and MIDI loops now go through published immutable snapshots instead of touching live containers from the audio/MIDI thread:

  • Station::GetLoopTakeSnapshot() and LoopTake::GetMidiLoopSnapshot() publish weak-ptr snapshots, consumed by MidiRouter on the realtime path.
  • MidiLoop::_model is now an atomic<shared_ptr> with an acquire/release Model() accessor, fixing unsafe concurrent access.
  • LoopTake MIDI capture/playback paths are guarded by _midiCaptureMutex, with an unlocked _RecordMidiEventUnlocked() helper to avoid re-entrant locking.
  • VstEditorWindowManager's window list is guarded by a mutex on its cold open/close/prune paths.

This pattern is now codified in AGENTS.md: prefer the existing published immutable snapshot pattern for cross-thread container access.

04Editor lifecycle at a glance

PreInit() UI thread LoadLibrary + VSTPluginMain + effOpen [GlContextScope] ↓ Load() job thread effSetSampleRate / effSetBlockSize / effMainsChanged(resume) ↓ OpenEditor(frame) UI thread effEditOpen(frame HWND) + effEditGetRect [GlContextScope] ↓ idle loop UI thread WM_TIMER 20ms → effEditIdle [GlContextScope] audioMaster{Idle,UpdateDisplay} → MessageVst2Idle ↓ CloseEditor() UI thread effEditClose [GlContextScope]

05Cleanup pass this commit

Keep the fixes; drop the development scaffolding.

  • Removed all [SHUTDOWN TRACE] console spam from Main.cpp, keeping the real shutdown sequence intact.
  • Fixed a latent automation bug: audioMasterBeginEdit / EndEdit were publishing a spurious 0.0f parameter value. Because EndEdit fires last, it clobbered the real value recorded by audioMasterAutomate. They now acknowledge only.
  • Removed two stale design docs (vst2-editor-fix-plan.md, vst2-editor-unresponsive-history.md) that described intermediate approaches — _uiThreadId, Prefetch process level, MessageVst2SizeWindowSync, _isEditorOpening, the WH_CALLWNDPROCRET hook, [VST2 TRACE] logging — none of which exist in the final code.
  • Kept vst2-editor-repaint-root-cause.md, the accurate technical record of the primary fix.
Regression watch: the two highest-risk things to never reintroduce are (1) instantiating the effect off the UI thread, and (2) removing the GlContextScope guards around UI-thread plugin dispatch. Either one brings back a frozen editor or a white app.

Verification: full solution build (Debug x64) green; native tests pass (192 passed, 1 device-dependent skip). Manual target: an animated plugin (Battery 4) repaints continuously for as long as the editor stays open, matching minihost.
