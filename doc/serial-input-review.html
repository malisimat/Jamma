Serial Input Landed Without Polluting the Audio Path.

The pedalboard packet stream is now decoded, routed through a dedicated serial reader, queued onto the scene job path, and matched against source-aware trigger bindings. Keyboard behavior stays intact, and rig JSON can now express serial transport explicitly.

What Changed

Implementation

  • user.serial now parses port, baudrate, and enabled.
  • Each trigger pair can declare source: "serial" or fall back to keyboard.
  • Trigger dispatch is now transport-neutral via source/value/state matching.
  • A dedicated serial reader decodes Arduino packets with header 0xA5.
  • Serial events are queued and pumped on the scene job path, not in the audio callback.
  • Parser logic now accepts button index 0 when the field is present.

Protocol

Pedalboard Wire Format

  • Header byte: 0xA5
  • Byte 2: button index 0..11
  • Byte 3: pressed flag 0 or 1
  • Decoder resynchronizes when a fresh header appears mid-packet.
  • Rig bindings can use button 0; that required removing the old zero-as-missing assumption.

Rubber Duck

Review Questions

Did we jam transport details into trigger state logic?

No. Trigger matching already had a source dimension; the fix was to expose it all the way from rig parsing through runtime ingress.

Did we accidentally touch the audio callback?

No. Serial I/O lives on its own reader thread and drains on the scene job path. The callback-owned hot-path list stayed clean.

What was the sneaky bug?

The parser treated numeric zero as “field absent”. That breaks serial pedal index 0, so presence flags now decide validity instead.

What is still unproven?

Actual device handshake behavior on a live Arduino over COM has not been exercised in this environment.

Residual Risk

What I Would Watch Next

  • Line settings: some Arduino boards reset on serial-open depending on DTR behavior.
  • Rig ergonomics: serial bindings are explicit and flexible, but there is not yet a higher-level preset format for pedal maps.
  • Operational logging: current logs are enough for debugging, but not yet tuned for a polished user-facing connection status display.
