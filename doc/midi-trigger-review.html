Verdict

Critical

Multiple MIDI callbacks can write into a single-producer queue

Why it matters: the new dedicated trigger-device callbacks and the main MIDI callback can all push into the same trigger ingress queue. That queue is explicitly implemented as SPSC. Once more than one producer is active, event loss, corruption, and reordering become possible under real controller use.

Evidence: the queue declares itself single-producer/single-consumer in JammaLib/src/engine/MidiQueue.h:11, while trigger events are pushed from both the shared main-MIDI path in JammaLib/src/engine/Scene.cpp:1081 and the per-device callbacks opened in JammaLib/src/engine/Scene.cpp:1295 and JammaLib/src/engine/Scene.cpp:1307.

High

Dedicated trigger devices are disabled by the global MIDI enable flag

Why it matters: rigs that use a separate MIDI pad or foot controller only for triggers stop working unless user.midi.enabled is also true. That couples trigger control to loop-recording MIDI enablement and contradicts the branch docs.

Evidence: trigger routes and dedicated trigger devices are prepared before the early return in JammaLib/src/engine/Scene.cpp:1288, but the function exits before the dedicated trigger-device open loop at JammaLib/src/engine/Scene.cpp:1305. The intended separate-device behavior is described in doc/README.md:112.

Medium

MIDI trigger routing no longer preserves first-consumer arbitration

Why it matters: key and serial trigger paths stop once a station eats the event, but the new MIDI route loop keeps going. Overlapping bindings can now fan a single note or CC event into multiple stations, which is a semantic drift from the existing trigger model.

Evidence: the MIDI loop in JammaLib/src/engine/Scene.cpp:1018 through 1026 never breaks on IsEaten, while the serial path does break at JammaLib/src/engine/Scene.cpp:1118 and the key path returns after the first consumer in JammaLib/src/engine/Scene.cpp:839 and 870.

Medium

Rig parsing fails open on invalid MIDI channel and ID data

Why it matters: malformed rigs do not fail safely. Invalid channels silently widen to match-any-channel, and large IDs are accepted before being masked down to 8 bits during binding encoding. That makes bad config harder to detect and easier to misroute live.

Evidence: parsing defaults to any-channel in JammaLib/src/io/RigFile.cpp:345, only narrows for channel 1..16 at JammaLib/src/io/RigFile.cpp:365, and later masks the ID in JammaLib/src/engine/Trigger.cpp:31.

Recommended Fix Order

  • Repair the trigger ingress topology first: either make the queue truly multi-producer safe or funnel all trigger-device callbacks through a single producer boundary.
  • Split trigger-device enablement from main MIDI loop-recording enablement so dedicated trigger devices can open even when the main MIDI input is disabled.
  • Decide whether unified trigger semantics should preserve first-consumer arbitration. If yes, the MIDI route loop needs to stop on consume and tests should lock that in.
  • Tighten rig-file validation for invalid channels and out-of-range IDs so malformed config is rejected instead of broadened.
