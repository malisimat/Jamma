Design Constraints

  • No replacement of existing MIDI loop-recording code.
  • Minimal logging only for device lifecycle, drops, and eaten trigger events.
  • Real-time audio path stays isolated from new trigger-device work.
  • Rig-file channel values remain one-based for the user.

Behavior Notes

  • Note On = press.
  • Note Off or Note On velocity 0 = release.
  • CC value > 0 = press.
  • CC value 0 = release.
  • Velocity is ignored for trigger decisions.

Execution Path

1. Parse Rig-file parsing resolves the optional MIDI trigger binding and converts channels to zero-based internal values.
2. Bind Trigger::FromFile converts that config into internal trigger bindings, the same shape used for existing trigger transitions.
3. Route Scene opens shared or dedicated MIDI input devices and pushes trigger-device events onto a separate ingress queue.
4. Apply On the job thread, matching triggers receive OnMidiEvent and run through the same debounce and state-machine code as other trigger sources.

Rubber Duck Review

Why not splice trigger logic into MIDI loop recording?

Because that would have coupled two behaviors the plan explicitly wanted separated. The final shape keeps performance MIDI ingress intact and adds trigger routing alongside it.

Why add MIDI bindings inside Trigger instead of a second state machine?

The debounce, press/release semantics, and action sequencing were already correct there. Reusing that logic cut regression risk and kept the feature focused.

What is the main residual risk?

Device-name matching is still string-based, because that is how the current MIDI device layer selects ports. If port names drift across machines, the config may need adjustment.

What would be the next hardening step?

Add a small scene-level routing test harness that injects trigger-device ingress events directly, so shared-device and dedicated-device behavior can be exercised without physical hardware.
