Overdub Timing Investigation Summary

Repository: Jamma | Branch: bugfix/overdub-timing-2 | Date: 2026-05-17

Bottom line:

The strongest reproduced cause of large real-world misalignment is quantised overdub end timing in QUANTISE_POWER mode combined with non-boundary stop timing, plus a test waveform periodicity that can mask absolute delay.

Existing alignment test passes Live symptom reproduced in tests Absolute lag measured at 8192 samples (16 blocks @ 512)

Hypotheses Checked

Hypothesis Verdict Evidence
Existing test does not represent real-world overdub timing. Supported Existing test uses LatencyIn=0, LatencyOut=0, PreDelay=0, no station clock quantisation, and trigger debounce disabled. Live app path uses station clock and runtime-trigger behavior.
Test is accidentally measuring only overdub loop (not source + overdub together). Mostly ruled out Added explicit route/mute-isolation test confirms both source and target are present when measured (after output settle).
Something in real app flow is missing in test and causes delay. Supported Quantisation path in Station::OnAction(TRIGGER_OVERDUB_END) with Timer::QUANTISE_POWER can produce large correction error, and with non-boundary end offset this produced large absolute lag in new tests.
Current parameters are lucky (block-aligned and too narrow). Supported Expanded parameter tests with EndTriggerOffsetBlocks = +/-1, +/-8 show measurable lag; offset 0 remains aligned.

Important Technical Finding

The original seed pattern in overdub tests repeats every 2000 samples. That periodicity can alias large true shifts into small apparent lags.

Example: a true shift of 4096 samples can appear as residual 96 samples against a 2000-sample periodic signal.

To break this aliasing, a non-periodic seed probe was added. With that probe plus power quantisation and non-boundary stop, measured absolute lag became: 

[OverdubTimingInvestigation] uniqueSeed powerQuantisedLag=8192 mse=0 samples=16384

That is 16 blocks at 512 samples/block, matching the “many blocks” symptom.

Likely Root Cause Chain

  1. Overdub ends slightly off loop boundary (human timing and/or trigger processing timing).
  2. Clock quantisation in QUANTISE_POWER mode snaps loop length and produces large error correction (observed +/-4096 sample cases).
  3. Playback start correction then shifts overdub take phase relative to source.
  4. Periodic test data can hide this in baseline test, so test passes while app can sound delayed by many blocks.

Suggested Next Actions

  1. Decide intended behavior for overdub end when quantising: QUANTISE_POWER vs QUANTISE_MULTIPLE for overdub specifically.
  2. Use non-periodic seed data in the core alignment test (or add an additional non-periodic strict test) to prevent alias-masked false passes.
  3. Add a dedicated app-like test path with realistic trigger timings and configured clock quantisation defaults.
