VST editor UI-thread freeze — diagnosed & fixed

Pressing Ctrl-Shift-E over a station with a loaded VST3 used to hard-freeze the entire host. After eight build/run/diagnose iterations driven by live telemetry files, the host UI now stays at ~60 fps with audio + jobs flowing, even while a misbehaving plug-in deliberately blocks during IPlugView::attached().

Root cause

The original code created the VST editor window and called IPlugView::attached(HWND) directly on the host's main render thread. Many VST3 plug-ins (notably VSTGUI-based ones such as ValhallaSupermassive) do internal SendMessage / COM marshalling round-trips inside attached(), requiring the HWND-owning thread to keep pumping messages while attached() runs. With one thread doing both, the plug-in waits for messages that the only thread that can dispatch them is busy waiting for the plug-in — classic deadlock.

The audio thread also held a separate hazard: IAudioProcessor::process() was running concurrently with attached(), and some plug-ins serialize their internal state machine across both, increasing deadlock risk.

The fix (three layered changes)

  1. Dedicated VST UI thread. VstEditorWindow now spawns its own thread that calls OleInitialize (STA), creates the editor frame HWND, runs a GetMessage loop, and tears everything down on close. The main render thread never touches the editor HWND.
  2. Pump-while-attaching. Inside the VST UI thread, OpenEditor() is dispatched to a short-lived worker thread. The HWND owner waits with MsgWaitForMultipleObjectsEx(QS_ALLINPUT | MWMO_INPUTAVAILABLE) and drains messages every wake-up, so any SendMessage the plug-in fires cross-thread completes immediately.
  3. Pause processing during open. A new std::atomic<bool> _editorOpening in VstPlugin short-circuits ProcessBlock / ProcessBlockStereo while the editor is being attached, removing audio↔GUI re-entrancy.

Result: Process.Responding == True for the entire session, render heartbeat / audio counters / job ticks all climb linearly, even when a plug-in chooses never to return from attached().

Lessons captured to repo memory

  • VST3 editor must run on its own dedicated UI thread — never call IPlugView::attached() on the host render thread.
  • Call attached() on a worker thread while the HWND-owning thread pumps via MsgWaitForMultipleObjectsEx.
  • Pause processor->process() while opening the editor (atomic flag checked in ProcessBlock).
  • VstEditorWindow lives in JammaLib (static lib): always build JammaLib.vcxproj before Jamma.vcxproj, otherwise stale code stays linked.

Known follow-up

Valhalla Supermassive's own attached() still does not return after this fix — but the host stays fully responsive while it spins. This is plug-in-internal behavior (likely a missing host interface such as IPlugInterfaceSupport or a DPI-awareness mismatch) and is out of scope for the UI-freeze bug. Tracked separately as a plug-in compatibility task.
