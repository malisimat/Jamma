Unified Trigger Interface

Two independent reviewers hit this branch from opposite angles: one on architecture and API shape, one on MIDI and serial trigger correctness. They converged on the same two core defects. The branch now fixes the ingress concurrency bug, restores cross-trigger fanout for matching key, serial, and MIDI events, and rejects malformed MIDI trigger bindings; the only remaining follow-up in this review is deferred multi-device MIDI support.

Verdict

Fixed

Single-callback MIDI trigger fanout now owns ingress

What changed: the branch no longer lets multiple MIDI callbacks write into a Scene-owned SPSC queue. The main MIDI input is now the single live MIDI source for both loop recording and MIDI trigger routing, and shared-main trigger fanout happens on the job thread when _midiIngress is drained.

Why it is fixed: there is now only one callback-owned MIDI ingress path, trigger routes all share the same main-MIDI slot, and the simplified single-global behavior is locked in by focused trigger regression coverage.

Multiple simultaneous MIDI devices are deferred to issue #91

Current scope: the branch now intentionally assumes one global MIDI input for both loop MIDI recording and MIDI triggers. That keeps device ownership, ingress topology, and routing semantics aligned while the rest of the feature settles.

Follow-up: proper multi-device support should return only when recording and trigger paths can both carry device identity cleanly. That work is now tracked separately in GitHub issue #91.

Fixed

Matching trigger events now fan out across all bound triggers

What changed: matching key, serial, and MIDI trigger events now continue through every bound trigger instead of stopping at the first eaten station or route.

Why it is fixed: overlapping bindings on the same key, serial input, or MIDI event now all fire, and focused regression coverage locks in fanout for key, serial, and MIDI plus the unmatched-event no-op behavior.

Fixed

Rig parsing now fails closed on invalid MIDI channel and ID data

What changed: MIDI trigger JSON now rejects explicit channels outside 1..16 and IDs above 127, and trigger construction defensively revalidates binding specs before encoding them.

Why it is fixed: malformed rigs now fail closed instead of widening to match-any-channel or being silently truncated during binding encoding, and dedicated parser regressions cover invalid channel and ID inputs.

Why The Review Is High Confidence

  • The architecture/API reviewer and the trigger-correctness reviewer independently landed on the same gating regression and the same routing/fanout concern, and both now have focused regression coverage.
  • The concurrency defect was structural, not stylistic: the queue contract and callback topology disagreed in plain code, and the repair now enforces a single callback producer with job-thread fanout.
  • Trigger fanout is now consistent across the station key path, serial ingress path, and shared-main MIDI route loop, and malformed MIDI trigger config is rejected before binding encoding with a second defensive check at trigger construction.
