Design notes

Storage
Fixed std::array<MidiEvent, 4096> per loop; no growth in RT path.
Capacity policy
Drop-newest on record overflow (loop is what it is); drop-oldest on ingress queue overflow (newest events always reach the consumer).
Held-note tracking
std::bitset<16×128> (256 B). Flushed on wrap and on StartRecord / EndRecord / Reset.
Block semantics
Half-open: an event at sample S fires in the block [S, S+N), never the previous one.
Time mapping
loopOffset = globalSample % loopLength; sink receives globally-rebased sampleOffset.
Quantisation
MidiLoop has zero Timer coupling; callers pass the snapped length, mirroring audio Loop’s convention.

Risks & open questions

  • Linear event scan in ReadBlock is O(n) per block. Fine for typical loop sizes; revisit if loops grow beyond a few thousand events.
  • Drop-newest on record matches the audio philosophy but should be visible in UI (counter surfaced as DroppedEventCount()).
  • Held-note flush on EndRecord currently resets the bitset without emitting NoteOffs; correct because Recording doesn’t drive a sink, but worth a comment when Slice 7 adds live monitoring.
  • Station seam (Slice 6) still TBD. Needs a minimal IMidiTake-style abstraction so callers don’t branch on media type in the hot path.
