Trigger schema — two responsibilities
🎛 Activation Sources

These fields define what physical event starts or stops the trigger — i.e., begins or ends a recording pass. Nothing here is recorded into the loop itself.

⌨ Keyboard — pairs[] (no source field)
activatedown Key code that starts recording on press
activateup Key code that confirms start on release
ditchdown / ditchup Key codes that abort/end recording
Default source when source is absent. Values are raw keyboard scan codes.
🔌 Serial pedalboard — pairs[] with "source": "serial"
source "serial"
device Must match a name in user.serial.devices[]
activatedown / activateup Button index on the serial device
ditchdown / ditchup Button index that ditches the take
Serial devices send simple integer button indices over a COM port at the configured baud rate.
🎹 MIDI note — pairs[] with "source": "midi"
source "midi"
device Must match a name in user.midi.devices[]
activatedown / activateup MIDI note number for start (NoteOn / NoteOff)
ditchdown / ditchup MIDI note number for ditch
🎛 Primary MIDI binding — trigger object
trigger.device MIDI device name for this primary binding
trigger.activate.kind "note" or "cc"
trigger.activate.channel MIDI channel (1–16)
trigger.activate.id Note number or CC number
trigger.ditch Same shape as activate
The trigger object registers a dedicated MIDI device route in the engine so its activate/ditch messages are dispatched at sample-accurate timing. A trigger can have both pairs bindings and a trigger object — any of them can fire the trigger.
🔴 Recording Sources

These fields define what content gets captured once the trigger is active — audio channels, MIDI notes, and which devices can play live through the station's VSTi.

🎙 Audio capture — input[]
input Array of ADC channel indices (0-based) to record
Each index corresponds to a physical input channel on the audio interface. Channels are latency-compensated and written into audio loop buffers on the station. An empty array [] means no audio is captured (MIDI-only station).
🎵 MIDI loop recording — midiinput[]
midiinput MIDI channels to record (1-based, e.g. [1] = ch 1)
Each listed MIDI channel gets its own MidiLoop inside the LoopTake. When the trigger is active, NoteOn/NoteOff events on these channels are timestamped and written into the loop. On playback, the loop replays them into the station's VST chain.
🎛 VST live playback & MIDI record filter — midiinputdevices[]
midiinputdevices Device names allowed to drive this station's VSTi
Two effects in one field:
1. Live VSTi playback — only MIDI events from the listed devices are passed to EnqueueLiveMidiEvent and heard in real time through the instrument plugin.
2. MIDI loop recording — RecordMidiEvent also filters by device name, so only notes from the listed devices are written into the MIDI loop.

If omitted or empty, all connected MIDI devices are allowed (open/backward-compatible). This is the key field for preventing a MIDI pedal on the trigger device from inadvertently playing through the VSTi.
Interaction between activation & recording

A MIDI device can appear in both roles simultaneously. For example, a MIDI pedal on Focusrite USB MIDI that sends note 3 to activate the trigger will also send that note through the live MIDI path unless you restrict midiinputdevices to a different controller. The midiinputdevices filter was designed exactly for this separation.
